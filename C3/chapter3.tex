This chapter outlines the mathematical formulations of the two ILP models and the software architecture designed to evaluate them.

\section{Specification and Requirements}\label{sec:specification-and-requirements}
Before the modelling or coding of the system could begin, the rules of the puzzle and evaluation goals had to be formally defined.

\subsection{Formal Problem Definition}\label{subsec:formal-problem-definition}

Battleship Solitaire is a single-player, deterministic logic puzzle played on a grid that represents an ocean. The objective is to discover the locations of the fleet of hidden ships.

The solver is provided with row and column tallies which represent how many individual ship segments occupy that specific line. Additionally, a small set of hints (pre-revealed cells) is provided to anchor the search. 
 
To solve the puzzle, the entire fleet must be placed onto the grid, such that all tallies are satisfied and no two ships touch each other, even diagonally. Figure~\ref{fig:battleship_example} illustrates the transformation from an initial problem state to fully solved, feasible grid.

The puzzle has many parameters that the solving algorithms have to process.

\textbf{The Grid:} The puzzle is played on a square grid of size $N \times N$, which consists of cells defined by their position using row $i$ and column $j$ coordinates.

\textbf{The Fleet ($F$):} A predefined set of ships that must all be placed on the board to complete the puzzle. The standard CSPLib fleet consists of a single Battleship (length 4), two Cruisers (length 3), three Destroyers (length 2) and four Submarines (length 1), the generated dataset fleet scales based off this.

\textbf{The Tallies ($R$ and $C$):} Each row $i$ has a required sum $R_i$, and each column $j$ has a required sum $C_j$. These numbers represent the total number of ship segments that are present in that specific row or column.

\textbf{The Hints ($H$):} A set of cells revealed prior to solving that act as a base to build from. Each hint provides a coordinate ($i, j$) and a specific state from the following set:
    \begin{itemize}
        \item 'w': Water
        \item 'c': Submarine
        \item 't', 'b', 'l', 'r': Top, Bottom, Left or Right ends of a multi-cell ship
        \item 'm': Middle segment of a ship (length of 3 or larger)
    \end{itemize}

\subsection{Logical Constraints (The Game Rules)}\label{subsec:logical-constraints}
There are rules that the standard Battleship Solitaire game sticks to that also must be processed by the algorithms.

\textbf{Tally Satisfaction:} The total number of ship segments in any row or column must match the corresponding tally.

\textbf{Fleet Satisfaction:} The entire fleet specified in $F$ must be placed onto the grid.

\textbf{Ship Geometry:} Ships must be linear (horizontal or vertically) blocks of continuous segments. They cannot be placed diagonally.

\textbf{Non-Adjacency:} Ships cannot touch each other. They must be entirely surrounded by water. This also blocks diagonal touches.

\textbf{Hint Adherence:} The completed board state must respect the initial hints provided in $H$.


\subsection{Software System Requirements}\label{subsec:software-sys-requirements}
These game rules were converted into software requirements, both functional and non-functional.

\textbf{Functional Requirements}
\begin{itemize}
    \item \textbf{Parser Integration:} The system must be able to read and process the Prolog format used in the Constraint Satisfaction Problem Library (CSPLib) \citep{csplib:prob014}.
    \item \textbf{Solver Implementations:} The system must implement the two optimisation models using the Gurobi interface. One being the Cell-based method which uses Big-M boundary logic, and the other being the Ship-based model which uses candidate generation and set packing.
    \item \textbf{Dataset Generation:} As the existing datasets are limited, the system must have a generator capable of building valid puzzle instances at a varying number of hints and and dimensions.
    \item \textbf{Evaluation and Benchmarking:} The system must feature an evaluator that can run the two algorithms sequentially, record their execution times and output visualisations to aid analysis.
    \item \textbf{Graphical Interface:} The system must provide a GUI that allows users to interactively input parameters and cell information, and visualise the solver's output.
\end{itemize}

\textbf{Non-Functional Requirements}
\begin{itemize}
    \item \textbf{Scalability:} The system must be capable of dealing with combinatorial explosion, managing extreme memory limits and time limits well.
    \item \textbf{Thread Safety and Responsiveness:} The graphical user interface must remain fully responsive during intensive evaluation of the Gurobi optimiser.
    \item \textbf{Reproducibility:} Evaluation outputs must be deterministic, so they can be reproduced if needed.
\end{itemize}

\section{System Architecture}\label{sec:system-architecture}

The software was designed to be modular from the beginning. The mathematical optimisation engines are decoupled from the user interface and the evaluation pipelines, so things can run independently if needed.

\subsection{Hardware and Execution Environment}\label{subsec:hardware-execution-env}

Computational complexity (NP-Hardness) is bound by physical hardware limits, so first the hardware environment will be defined.

The development, dataset generation and evaluation runs were executed on a 2024 Apple Macbook Air, using the M3 chip and 16GB of unified memory.

The M3 provides high single-thread performance, which benefits Gurobi's branch-and-bound search tree used in the Cell model. The 16GB of memory will be the main limitation for the Ship model's candidate generation, once the grid reaches larger sizes (e.g. $50 \times 50$ and higher).

The operating system of the device is macOS Sequoia 15.7.4, which features strict thread-safety protocols.

\subsection{Core System Modules}\label{subsec:core-sys-modules}

The architecture follows a layered design that separates concerns across four cooperating components, with thread-safe message passing between the optimisation back-end and the user interface front-end. The four layers are described in turn below.

\begin{enumerate}[label=\textbf{O\arabic*:}, leftmargin=*]
    \item \textbf{The Data Ingestion Layer:} This layer is responsible for following the standardised data format. It contains the Regex pipeline that reads the standard Prolog CSPLib files. It also contains the custom puzzle generator used for scalability testing on the models. This layer packages the information on the provided standardised puzzle into a \verb|BattleshipPuzzle| Python object. This ensures that both ILP solvers receive identical data structures.
    \item \textbf{The Optimisation Engine:} This layer works by calling \verb|.solve(puzzle)|, which swaps between the Cell model and Ship model without needing to alter the underlying code. The solvers in turn translate the \verb|BattleshipPuzzle| object into matrix equations and feed them into the Gurobi Optimiser \citep{gurobi2024}, which executes the linear algebra and returns the solved matrix to the Python wrapper.
    \item \textbf{The Presentation Layer:} This layer is built using CustomTkinter, and provides the interactive visual single-run grid and batch-evaluation controls. Gurobi solver executions are run on asynchronous background threads to prevent the GUI from freezing. A \verb|textboxRedirector| is used to intercept the terminal output and queue it for the main UI using event scheduling.
    \item \textbf{The Evaluation Layer:} This module automates the generation of statistical evidence. It calculates solver speeds and shows them in comparison to puzzle difficulty (categorised by number of hints). Matplotlib is run headless to prevent macOS thread violations during batch runs, with the results being saved as \verb|.png| files.
\end{enumerate}

\subsection{System Data Flow}\label{subsec:sys-data-flow}

A quick step-by-step summary on how data moves through the system during a single-puzzle solve using the GUI.
\begin{enumerate}[label=\textbf{O\arabic*:}, leftmargin=*]
    \item \textbf{Input:} The user clicks grid cells to input the hints and types the corresponding tallies into the CustomTkinter GUI.
    \item \textbf{Translation:} The GUI compiles the visual input into a formatted CSPLib string and instantiates a BattleshipPuzzle object.
    \item \textbf{Dispatch:} The GUI passes the object to the selected solver on a background thread.
    \item \textbf{Optimisation:} Gurobi calculates the optimal candidate and returns the geometric coordinates to Python.
    \item \textbf{Update:} The main thread reads the given coordinates and updates the cell on the GUI to reveal the solution to the user.
\end{enumerate}

\section{Mathematical Formulation: The Cell-Based Model}\label{sec:math-formulation-cell}

The Cell-based approach models the puzzles by treating the battleship grid as a matrix of individual squares. The solver then determines the state of each cell and applies localised boundary logic to ensure the fleet is formed correctly without breaking any of the game rules. The constraint equations are from Meuffels \citep{meuffels-battleship}.

\subsection{Decision Variables}\label{subsec:cell-decision-variables}
To capture the presence of a ship in a cell, and its specific shape, the cell-model uses a set of binary decision variables for every cell $(i, j)$ in the $N \times N$ grid.

Let $S$ be the set of all possible cell states: $S = \{w, c, t, b, l, r, m\}$ representing Water, Circle (submarine), Top, Bottom, Left, Right and Middle respectively.

We define the binary variable for each state in each cell as:
\begin{equation}
    v_{i,j}^s \in \{0, 1\} \quad \forall i,j \in \{1 \dots N\}, \forall s \in S
\end{equation}

Each cell must have exactly one state at any given time, so a uniqueness constraint is applied:
\begin{equation}
    \sum_{s \in S} v_{i,j}^s = 1 \quad \forall i,j \in \{1 \dots N\}
\end{equation}

For convenience when calculating the row/column tallies, we define an auxiliary binary variables $\sigma_{i,j}$ which equals 1 if the cell contains any form of ship segment, and 0 if it is water:
\begin{equation}
    \sigma_{i,j} = 1 - v_{i,j}^w
\end{equation}

\subsection{Tally Constraints}\label{subsec:tally-constraints}

The most fundamental rule is the requirements of the sum of the ship segments in any row or column to perfectly match the provided tallies, $R_i$ and $C_j$. This is formulated as straight-forward linear summations:

\textbf{Row Tallies:}
\begin{equation}
    \sum_{j=1}^{N} \sigma_{i,j} = R_i \quad \forall i \in \{1 \dots N\}
\end{equation}

\textbf{Column Tallies:}
\begin{equation}
    \sum_{i=1}^{N} \sigma_{i,j} = C_j \quad \forall j \in \{1 \dots N\}
\end{equation}

\subsection{Geometric and Non-Adjacency Constraints}\label{subsec:geo-and-non-adj-constraints}

These constraints are what cause the Cell Model to become computationally expensive. The solver must use dense conditional logic to prevent any ship from touching.

\textbf{Diagonal Non-Adjacency:}
Ships are forbidden from touching diagonally. Therefore, for any $2 \times 2$ block of cells, two diagonally opposite cells cannot both contain a ship segment simultaneously:
\begin{equation}
    \sigma_{i,j} + \sigma_{i+1, j+1} \leq 1 
\end{equation}
\begin{equation}
    \sigma_{i+1, j} + \sigma_{i, j+1} \leq 1
\end{equation}

\textbf{Shape Transition Logic:}
To ensure that ships form valid linear blocks, the state of one cell strictly dictates the allowed states of its neighbour. For example, if a cell is designated as a Top segment (\textit{t}), the cell immediately below it must be either a Bottom (\textit{b}) or a Middle (\textit{m}). This implication logic is modelled as a linear inequality:
\begin{equation}\label{eq:cell-shape-transition}
    v_{i,j}^t \leq v_{i+1, j}^m + v_{i+1, j}^b
\end{equation}
The same pattern is applied to the other three end states. For any cell $(i,j)$:
\begin{align}
    v_{i,j}^l &\leq v_{i, j+1}^m + v_{i, j+1}^r \\
    v_{i,j}^r &\leq v_{i, j-1}^m + v_{i, j-1}^l \\
    v_{i,j}^b &\leq v_{i-1, j}^m + v_{i-1, j}^t
\end{align}
At the grid boundary, end pieces facing off-grid are forbidden:
\begin{equation}
    v_{i,N}^l = v_{i,1}^r = v_{N,j}^t = v_{1,j}^b = 0 \quad \forall i, j \in \{1, \dots, N\}
\end{equation}

\subsection{Fleet Inventory and Pattern Matching}\label{subsec:fleet-inventory}

The constraints in Sections \ref{subsec:tally-constraints} and \ref{subsec:geo-and-non-adj-constraints} guarantee that every cell holds a single state and that ship segments form valid lines. They do not, however, guarantee that the resulting fleet matches the specification $F$. Two further types of constraints are required: a global head-count constraint, and a per-length pattern matcher.

\textbf{Total fleet count.}
Every ship in the fleet has exactly one ``head'' segment: a single-cell ship is a submarine ($c$) and is itself a head; a horizontal ship of length two or more contains exactly one left-end ($l$); a vertical ship of length two or more contains exactly one top-end ($t$). Counting heads therefore counts ships, and the total must equal $|F| = \sum_{L \in F} F_L$, the cardinality of the fleet:
\begin{equation}\label{eq:cell-fleet-count}
    \sum_{i=1}^{N} \sum_{j=1}^{N} \left( v_{i,j}^c + v_{i,j}^l + v_{i,j}^t \right) = |F|
\end{equation}

\textbf{Per-length pattern matching.}
The total head count alone does not fix how many ships of each length appear. To enforce the per-length counts $F_L$, an auxiliary binary detector $z_{L,o,i,j} \in \{0, 1\}$ is introduced for every length $L$, orientation $o \in \{H, V\}$, and starting cell $(i,j)$ at which a ship of that length and orientation could fit. The detector takes the value $1$ if and only if the corresponding placement is realised by the cell-state variables.

For length $L = 1$ (submarines), the count is direct, with no detector needed:
\begin{equation}
    \sum_{i=1}^{N} \sum_{j=1}^{N} v_{i,j}^c = F_1
\end{equation}

For $L > 1$, define the placement set $P_H(L, i, j) = \{(i, j, l)\} \cup \{(i, j + L - 1, r)\} \cup \{(i, j + k, m) : 1 \leq k \leq L - 2\}$ for the horizontal orientation, and analogously $P_V(L, i, j) = \{(i, j, t)\} \cup \{(i + L - 1, j, b)\} \cup \{(i + k, j, m) : 1 \leq k \leq L - 2\}$ for the vertical orientation. The detector $z_{L,o,i,j}$ must equal the logical AND of the $L$ state variables in $P_o(L, i, j)$. This AND is linearised in the standard way:
\begin{equation}
    z_{L,o,i,j} \leq v_{i', j'}^{s'} \quad \forall (i', j', s') \in P_o(L, i, j)
\end{equation}
\begin{equation}
    z_{L,o,i,j} \geq \sum_{(i', j', s') \in P_o(L, i, j)} v_{i', j'}^{s'} \;-\; (L - 1)
\end{equation}
The first family forces $z_{L,o,i,j} = 0$ whenever any required piece is absent. The second forces $z_{L,o,i,j} = 1$ when all $L$ required pieces are present.

The per-length count is then expressed as:
\begin{equation}\label{eq:cell-per-length-count}
    \sum_{(i,j)} z_{L,H,i,j} + \sum_{(i,j)} z_{L,V,i,j} = F_L \quad \forall L \in F, \; L > 1
\end{equation}
where the inner sums range over the valid starting positions for each orientation (i.e. positions for which the ship fits inside the grid).

The cost of this formulation is the introduction of one auxiliary binary detector per (length, orientation, starting position), each with $L + 1$ accompanying linearisation constraints. For an $N \times N$ grid and a fleet containing ship lengths up to $L_{\max}$, this adds $\mathcal{O}(N^2 L_{\max})$ detector variables and $\mathcal{O}(N^2 L_{\max}^2)$ linearisation constraints. This contributes to the dense matrix structure that Chapter~\ref{ch4} identifies as the dominant factor in the Cell Model's solve time at scale.

\subsection{Linearised Degree Constraints}\label{subsec:degree-constraints}

The shape-transition constraints in Section~\ref{subsec:geo-and-non-adj-constraints} forbid invalid neighbour states locally, but they do not directly enforce the degree of each cell (how many orthogonal neighbours must be ship segments). A submarine must have zero ship neighbours, an end piece must have exactly one, and a middle piece must have exactly two. These rules are enforced through Big-M linearised equalities, where the constant attached to the slack term is chosen as the smallest value that renders the constraint inactive when the corresponding indicator is zero.

\textbf{Neighbour Count.}
For each cell $(i, j)$, let $\mathcal{N}(i, j)$ denote the set of orthogonal in-bounds neighbour coordinates. Reusing the auxiliary $\sigma_{i,j}$ from Section~\ref{subsec:cell-decision-variables}, define:
\begin{equation}\label{eq:cell-eta-def}
    \eta_{i,j} = \sum_{(i', j') \in \mathcal{N}(i,j)} \sigma_{i', j'}
\end{equation}
$\eta_{i,j}$ is the number of orthogonal neighbours of $(i, j)$ that contain a ship segment, and is bounded above by $|\mathcal{N}(i, j)| \leq 4$.

\textbf{Submarine (Degree 0).}
A submarine is an isolated single-cell ship. If $v_{i,j}^c = 1$, then $\eta_{i,j}$ must equal $0$:
\begin{equation}\label{eq:cell-degree-sub}
    \eta_{i,j} \leq 4 \cdot (1 - v_{i,j}^c)
\end{equation}
When $v_{i,j}^c = 1$, the right-hand side collapses to $0$ and forces $\eta_{i,j} = 0$. When $v_{i,j}^c = 0$, the right-hand side becomes $4$ and the constraint is trivially satisfied (since $\eta_{i,j} \leq 4$ is already implied). The constant $4$ is the Big-M value, chosen as the maximum number of orthogonal neighbours any cell can have.

\textbf{End Pieces (Degree 1).}
End pieces ($l, r, t, b$) lie at the boundary of a multi-cell ship and must have exactly one ship neighbour --- the next segment of their own ship. Let $e_{i,j} = v_{i,j}^l + v_{i,j}^r + v_{i,j}^t + v_{i,j}^b$ be the indicator that $(i, j)$ is an end piece. The exactly-one-neighbour rule is enforced by sandwiching $\eta_{i,j}$:
\begin{equation}
    \eta_{i,j} \leq 1 + 3 \cdot (1 - e_{i,j})
\end{equation}
\begin{equation}
    \eta_{i,j} \geq 1 - (1 - e_{i,j})
\end{equation}
When $e_{i,j} = 1$, both bounds collapse to $\eta_{i,j} = 1$. When $e_{i,j} = 0$, the upper bound relaxes to $4$ and the lower bound to $0$, deactivating the constraint. The Big-M values $3$ (upper) and $1$ (lower) are the smallest constants that achieve full deactivation given the bound $0 \leq \eta_{i,j} \leq 4$.

\textbf{Middle (Degree 2).}
A middle piece ($m$) lies strictly inside a ship of length three or more and must have exactly two ship neighbours, one on either side along the ship's axis:
\begin{equation}
    \eta_{i,j} \leq 2 + 2 \cdot (1 - v_{i,j}^m)
\end{equation}
\begin{equation}
    \eta_{i,j} \geq 2 - 2 \cdot (1 - v_{i,j}^m)
\end{equation}
When $v_{i,j}^m = 1$, both bounds collapse to $\eta_{i,j} = 2$. When $v_{i,j}^m = 0$, the bounds relax to $4$ and $0$ respectively. The Big-M value of $2$ is sufficient on both sides because $\eta_{i,j}$ is bounded between $0$ and $4$ and the target value is $2$.

These six inequalities, applied at every cell of the grid, are the dominant source of constraint density in the Cell Model. Together with the pattern-matching detectors of Section~\ref{subsec:fleet-inventory} they account for the matrix bloat that Chapter~\ref{ch4} identifies as stalling Gurobi's root-node Linear Programming relaxation, even on instances that would otherwise be solved without any branching.

\subsection{Hint Adherence}\label{subsec:cell-hint-adherence}

Any pre-revealed hints provided in the set $H$ are enforced through bounding the corresponding decision variable to 1, instantly pruning invalid branches from the solver's search tree. If hint $H$ dictates that cell ($i$, $j$) is state $s \in S$, that variable is fixed:
\begin{equation}
    v_{i,j}^s = 1
\end{equation}

\section{Mathematical Formulation: The Ship-Based Model}\label{sec:math-formulation-ship}

The ship-based solver works by generating every valid ship placement upfront and asking the solver to pick the correct combination of ships from the selection, instead of working cell-by-cell. The constraint equations are from Meuffels \citep{meuffels-battleship}.

\subsection{Pre-Processing and Candidate Generation}\label{subsec:pre-proc-and-candidate-gen}

Before the optimisation matrix is built, a pre-processing algorithm traverses the grid to generate the complete set of valid ship candidates, $\mathcal{C}$.

A candidate $c \in \mathcal{C}$ represents a ship of length L placed at specific coordinates. Candidates are pruned to reduce the search space before Integer Linear Programming (ILP) begins.

No candidate is generated on cells known to be water (from the hint set $H$), and no candidate is generated if it would intersect a row $i$ where $R_i = 0$, or a column $j$ where $C_j = 0$.

\subsection{Decision Variables}\label{subsec:ship-decision-variables}

The model only requires a single, one-dimensional array of binary decision variables as the geometry is pre-calculated.

Let $y_c \in \{0, 1\}$, 1 if candidate $c$ is selected for the final board state, and 0 otherwise:
\begin{equation}
    y_c \in \{0, 1\} \quad \forall c \in \mathcal{C}
\end{equation}

\subsection{Fleet and Tally Constraints}\label{subsec:fleet-and-tally-constraints}

The model must fulfill the exact fleet specification and the discrete tomography tallies provided.

\textbf{Fleet Constraint:}
Let $\mathcal{C}_k \subseteq \mathcal{C}$ be the subset of all candidates of a specific ship class $k$ (e.g. all of the Destroyers). The number of selected candidates from this class must match the required fleet count $F_k$.
\begin{equation}
    \sum_{c \in \mathcal{C}_k} y_c = F_k \quad \forall k \in \text{Classes}
\end{equation}

\textbf{Tally Constraints:}
Let $a_{i,c}$ represent the number of segments candidate $c$ occupies in row $i$, and $b_{j,c}$ represent the segments in column $j$. The sum of all segments from the chosen candidates must satisfy the tallies:
\begin{equation}\label{eq:ship-row-tally}
    \sum_{c \in \mathcal{C}} a_{i,c} y_c = R_i \quad \forall i \in \{1 \dots N\}
\end{equation}
\begin{equation}\label{eq:ship-col-tally}
    \sum_{c \in \mathcal{C}} b_{j,c} y_c = C_j \quad \forall j \in \{1 \dots N\}
\end{equation}

\subsection{Edge-Based Set Packing (Conflict Constraints)}\label{subsec:edge-based-set-packing}
This is where the Ship Model finds its speed, as Gurobi deals with sparse matrices with extreme efficiency. Due to the fact ships cannot overlap or touch diagonally/adjacently, many of the candidates are mutually exclusive.

Rather than using complex transition logic like the Cell Model, this model builds a Conflict Graph. This is formulated as a highly sparse Edge-Packing constraint. Let $E$ be the set of all edges representing a conflict between two candidates. Two candidates, $u$ and $v$, conflict if they overlap or touch:
\begin{equation}\label{eq:ship-edge-packing}
    y_u + y_v \leq 1 \quad \forall (u, v) \in E
\end{equation}

\subsection{Hint Adherence}\label{subsec:hint-adherence}
Hints are enforced by ensuring that exactly one candidate matching the hint is selected. Let $\mathcal{C}_h$ be the subset of candidates that place the exact correct ship segment over the hint coordinate $h \in H$:
\begin{equation}
    \sum_{c \in \mathcal{C}_h} y_c = 1 \quad \forall h \in H
\end{equation}

\section{Valid Inequalities and Model Optimisation}\label{sec:valid_inequalities}

Integer Linear Programming (ILP) solvers such as Gurobi, rely heavily on Branch-and-Bound algorithms to search solution space further from the logical constraints of the Battleship Puzzle.

Four valid inequalities were engineered to improve computational efficiency and reduce the number of nodes the solver must explore. They were implemented into an Improved Cell-Based Model.

These valid inequalities are mathematically redundant constraints as they do not alter the final feasible solution space, but they tighten the mathematical polytope to aid the solver.

The four valid inequalities below reuse the decision variables $v_{i,j}^s$ introduced in Section~\ref{subsec:cell-decision-variables}, with state set $S = \{w, c, t, b, l, r, m\}$ standing for water, submarine (circle), top, bottom, left, right and middle. The grid dimensions $N \times N$ and the row/column tallies $R_i$, $C_j$ from Section~\ref{subsec:tally-constraints} are also reused.

\subsection{Global Parity}\label{subsec:global-parity}
Every horizontal ship of length two or greater must have exactly one left-end piece and one right-end piece. Similarly, every vertical ship must have exactly one top and one bottom piece. Therefore, the absolute sum of left pieces across the entire board must strictly equal the sum of right pieces.
\begin{equation}
    \sum_{i=1}^{N} \sum_{j=1}^{N} v_{i,j}^l = \sum_{i=1}^{N} \sum_{j=1}^{N} v_{i,j}^r
\end{equation}
\begin{equation}
    \sum_{i=1}^{N} \sum_{j=1}^{N} v_{i,j}^t = \sum_{i=1}^{N} \sum_{j=1}^{N} v_{i,j}^b
\end{equation}

\subsection{Line-Specific Parity}\label{subsec:line-specific-parity}
Because ships cannot bend, a horizontal ship must be contained entirely within a single row. This allows the global parity logic to be scaled down to individual rows and columns, severely restricting the solver's branching options within localized areas.
\begin{equation}
    \sum_{j=1}^{N} v_{i,j}^l = \sum_{j=1}^{N} v_{i,j}^r \quad \forall i \in \{1, \dots, N\}
\end{equation}
\begin{equation}
    \sum_{i=1}^{N} v_{i,j}^t = \sum_{i=1}^{N} v_{i,j}^b \quad \forall j \in \{1, \dots, N\}
\end{equation}

\subsection{Tally-Based Impossibilities}\label{subsec:tally-based-impossibilities}
If a row's required ship segment tally is exactly one ($R_i = 1$), it cannot contain a horizontal ship end-piece (Left or Right), as placing either would imply a minimum horizontal ship length of two, thereby violating the tally limit. A symmetric logic applies to columns where $C_j = 1$.
\begin{equation}
    v_{i,j}^l = 0 \quad \text{and} \quad v_{i,j}^r = 0 \quad \forall j \in \{1, \dots, N\}, \text{ if } R_i = 1
\end{equation}
\begin{equation}
    v_{i,j}^t = 0 \quad \text{and} \quad v_{i,j}^b = 0 \quad \forall i \in \{1, \dots, N\}, \text{ if } C_j = 1
\end{equation}

\subsection{Corner Banning}\label{subsec:corner-banning}
A ``Middle" piece ($M$) strictly requires two adjacent ship pieces on opposite sides (either horizontally or vertically). The four absolute corners of the grid only possess two orthogonal neighbours, making it mathematically impossible for a middle piece to occupy a corner cell.
\begin{equation}\label{eq:vi-corner-banning}
    v_{1,1}^m = 0, \quad v_{1,N}^m = 0, \quad v_{N,1}^m = 0, \quad v_{N,N}^m = 0
\end{equation}

\section{Datasets and Dynamic Generation}\label{sec:datasets-dynamic-gen}

To evaluate the scalability and computational efficiency of the ILP models, a robust dataset was required. A hybrid approach was adopted because publicly available datasets are limited to standard grid sizes. A standard library was utilised for baseline validation, and a custom generator utilised for scalability testing.

\subsection{Standard Academic Dataset (CSPLib)}\label{subsec:standard-academic-dataset}

To ensure the solvers were mathematically sound and could replicate known academic benchmarks, the system was engineered to ingest data from the Constraint Satisfaction Problem Library (CSPLib), specifically Problem 014 (Solitaire Battleships) \citep{csplib:prob014}.

Standard CSPLib files are formatted as Prolog strings. A custom Regular Expression (Regex) pipeline was implemented to parse these strings, extracting the grid dimensions, fleet specification, row/column tallies and the provided hints.

The CSPLib database publishes puzzle instances but not their reference solutions, so validation in this project is feasibility-based rather than equivalence-based. After solving each instance, the returned configuration was passed through a standalone validator (\verb|BattleshipBoard.is_valid_solution|) that re-checks the row and column tallies, the fleet composition against $F$, and the non-adjacency rule including diagonal touches.

Every CSPLib instance solved by the system was independently confirmed to satisfy these constraints. For well-posed Battleship Solitaire puzzles the published instances are constructed to admit a unique solution, so satisfying every constraint is in practice equivalent to recovering the intended answer. This is, however, an inference about CSPLib's curation rather than a property the validator itself enforces.

\subsection{Procedural Puzzle Generation Algorithm}\label{subsec:procedural-puzzle-gen-algo}
The evaluation of the models required puzzles of varying dimensions to test against the limits of combinatorial expansion. Battleship Solitaire puzzles cannot simply be randomised, so an algorithm was engineered that builds the puzzles in reverse to ensure they are feasible.

\begin{enumerate}[label=\textbf{O\arabic*:}, leftmargin=*]
    \item An empty grid of size $N \times N$ is initialised in memory.
    \item The standard $10 \times 10$ fleet consists of 10 ships. For custom dimensions, the generator scales the fleet density to ensure the board maintains an appropriate constraint-to-variable ratio.
    \item The algorithm iteratively attempts to place each ship from the fleet onto the grid using a randomised coordinate and orientation. The generator checks the non-adjacency rules and rejects and regenerates coordinates if a placement causes two ships to touch.
    \item Once the fleet is placed, the algorithm calculates the ship segments to get the row and column tallies.
    \item  Finally, the algorithm randomly selects a predetermined number of cells to act as pre-revealed hints, then outputs the complete puzzle object for the solvers to evaluate.
\end{enumerate}

\subsection{Controlling Scalability and Heuristic Difficulty}\label{subsec:controlling-scalability-heuristic-diff}

Puzzles were generated at progressively larger dimensions (10x10, 15x15, 20x20, 25x25 and 30x30). This was critical for observing the exponential time complexity ($O(2^n)$) inherent to NP-Complete problems, in particular tracking when the Ship Model's memory requirements for candidate generation exceeds system limits.

The difficulty of a Solitaire Battleship instance is largely dictated by the number of starting hints. These hints act as anchors that prune the search tree. The less hints, the more the solver relies on the row/column tallies. The generator produced datasets categorised by hint density:
\begin{itemize}
    \item \textbf{Easy:} 10+ hints.
    \item \textbf{Medium:} 4--9 hints.
    \item \textbf{Hard:} 0--3 hints.
\end{itemize}

\section{Implementation}\label{sec:impl}
% Realisation: key implementation choices and critical code; avoid describing every module. Optional: tools, platform, problems encountered.

The models and architecture described in the previous sections were implemented using Python 3, chosen for its robust object-orientated capabilities and native integration with optimisation libraries. The core mathematical optimisation was handled by the \verb|gurobipy| API, interfacing directly with the Gurobi Optimiser backend. 

\subsection{Data Structures and Parsing}\label{subsec:data-structures-parsing}
A central \verb|BattleshipPuzzle| class was implemented to encapsulate the grid dimensions, fleet specification, tallies and hints. This prevented the passing of raw arrays between functions.

To ensure compatibility with standard academic datasets, a custom parsing module was implemented. The CSPLib datasets store problem instances as Prolog strings. A Regular Expression (Regex) pipeline was engineered (with syntax assistance from \citet{gemini}) to extract these parameters:
\begin{verbatim}
    pattern = re.compile(
        r'problem\s*\(\s*(\d+)\s*,'      
        r'\s*\[(.*?)\]\s*,'              
        r'\s*\[(.*?)\]\s*,'              
        r'\s*\[(.*?)\]\s*,'              
        r'\s*\[(.*?)\]\s*\)\.',          
        re.DOTALL
    )
\end{verbatim}
This decoupling ensured that regardless of how the problem was loaded, the Gurobi solvers received an identical, standardised object.

\subsection{Mapping ILP Constraints in Gurobi}\label{subsec:mapping-ilp-constraints-gurobi}
A key implementation challenge was accurately translating the formal mathematical equations into Gurobi's Python API \citep{gurobi2024}.

For the Cell Model, this involved instantiating a multi-dimensional array of binary decision variables using \verb|model.addVars()|, indexing them by row, column and cell state. The shape-transition rule of Equation~\eqref{eq:cell-shape-transition} maps directly on to a \verb|gurobipy| constraint:
\begin{verbatim}
    model.addConstr(v[r, c, 't'] <= v[r+1, c, 'b'] + v[r+1, c, 'm'])
\end{verbatim}

The Ship Model implementation heavily relied on Python's pre-processing capabilities. The search space was drastically reduced before Gurobi was invoked by generating a dictionary of valid candidates. Each Edge-Based Set Packing conflict, defined in Equation~\eqref{eq:ship-edge-packing}, maps to a single sparse mutual-exclusivity constraints:
\begin{verbatim}
    model.addConstr(y[u] + y[v] <= 1)
\end{verbatim}

\subsection{Platform Challenges: Thread Safety and Concurrency}\label{subsec:platform-challenges-thread-safety}
A notable engineering problem was encountered during the development of the CustomTkinter Graphical User Interface on macOS.

Due to the fact that the complex matrices used in solving the Solitaire Battleship instances are computationally intensive, executing the \verb|gurobipy| optimiser on the main application thread caused the interface to freeze. 

The standard solution of running the solver on an asynchronous background thread resulted in fatal system crashes (\verb|NSInternalInconsistencyException|). This occurred because macOS strictly prohibits background threads from directly modifying UI components or terminal outputs.

The solution that worked (identified through troubleshooting with \citet{gemini}) was implementing a thread-safe event queueing system. A \verb|TextboxRedirector| class was engineered to intercept the standard \verb|print()| statements from the background solver and safely schedule them to be rendered by the main thread using the \verb|tkinter.after()| event loop:
\begin{verbatim}
    def write(self, text):
        self.textbox.after(0, self._insert_text, text)
\end{verbatim} 

\subsection{Headless Evaluation}\label{subsec:headless-evaluation}
To facilitate the automated evaluation of hundreds of problems without triggering similar window manager exceptions, the Matplotlib visualisation library was configured to operate in 'headless' mode, with the use of \verb|matplotlib.use('Agg')|.

The system saves the plots to the device's storage, allowing much faster batch evaluations.

\section{Summary}\label{sec:summary}
% Optional: short summary of the methodology. Omit if the chapter is already concise.

This chapter has detailed the formal mathematical specification and software engineering required to benchmark the different Integer Linear Programming approaches to the NP-Complete Solitaire Battleship problem.

A complete decoupled software architecture was engineered to complete the comparative analysis. A procedural forward-generator and custom CSPLib parser were implemented to ensure the solvers could be evaluated against both standard academic baselines and dynamic scaling, zero-hint datasets.

Furthermore, engineering specific to macOS such as asynchronous thread-safe event queueing and headless visualisation was implemented. This was to guarantee a responsive and stable automated testing environment.

Two distinct optimisation paradigms were successfully formulated using the Gurobi Python API:
\begin{itemize}
    \item \textbf{The Cell-Based Model:} A localised, dense-matrix approach that relies on Big-M transition constraints to enforce valid ship geometry and prevent adjacency.
    \item \textbf{The Ship-Based Model:} A pre-processing, sparse-matrix approach that utilises candidate generation and Edge-based Set Packing to resolve geometric conflicts.
\end{itemize}

The following chapter will subject both models to empirical testing, using the constraints and evaluation pipeline defined.
