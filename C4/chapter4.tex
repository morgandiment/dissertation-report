\subsection{Hardware and Environment}\label{hardware-environment}
All evaluations were executed on a Macbook Air equipped with the Apple M3 processor and 16GB of RAM, running on macOS Sequoia 15.7.4. The underlying ILP solver utilised for all tests was the Gurobi Optimiser (Version 13.0.1). interfaced with the \verb|gurobipy| Python library.

\subsection{Evaluation Metrics}\label{evaluation-metrics}
Evaluating algorithmic efficiency solely using system time can be susceptible to hardware fluctuations and background operating system processes. To ensure a resilient evaluation, two main metrics were recorded for every puzzle instance:
\begin{itemize}
  \item \textbf{Solve Time (Seconds):} The total time required for Gurobi to return an optimal solution or declare the model infeasible.
  \item \textbf{Node Count:} The total number of Branch-and-Bound nodes explored by the Gurobi solver. This provides a deterministic, hardware-independent metric of mathematical complexity. A node count of zero shows the puzzle was solved purely through heuristics, while a high node count shows the solver was forced to systematically guess and backtrack.
\end{itemize}

\subsection{Datasets and Test Scenarios}\label{datasets-test-scenarios}
In order to thoroughly test the boundaries of the models, the evaluation was done in three main runs:
\begin{enumerate}
  \item \textbf{Cumulative and Scaling Performance:} A custom dataset of dynamically generated puzzles (\verb|scalable_puzzles.txt|) ranging from 5x5 to 15x15 grids was used to test how each model scales with exponentially increasing board sizes.
  \item \textbf{Difficulty and Density Impact:} A standardised dataset of 10x10 puzzles from CSPLib was evaluated to determine how hint density (categorised in to Easy, Medium and Hard) impacts solver efficiency on a fixed grid size.
  \item \textbf{Constraint Ablation:} To explicitly quantify the impact of the valid inequalities, the Improved Cell Model was subjected to an ablation study on Medium and Hard 10x10 instances, disabling one custom constraint at a time to measure its individual effect on the overall solve time.
\end{enumerate}

The empirical evaluation of the Integer Linear Programming models yielded definitive results regarding algorithmic efficiency, scalability limits, and the impact of polyhedral tightening via valid inequalities.

\subsection{Cumulative Performance and Difficulty Analysis}\label{cumulative-performance}
The standard 10x10 CSPLib dataset was processed to establish a baseline of solver efficiency across varying heuristic difficulties. As visualised in the scatter plot (Figure~\ref{fig:difficulty_scatter}), the Ship-based model has a computational dominance over the Cell-based model across all instances.

Notably, the performance gap widens significantly as the hint density decreases. For "Hard" instances (0-3 hints), the standard Cell model frequently requires over 0.4 seconds to resolve the dense transition logic. In contrast, the Set Packing geometry of the Ship model resolves the same instances in near-constant time, tightly clustered near the 0.05-second mark regardless of hint density.

This resilience against combinatorial explosion is further evidenced by the cumulative performance Cactus Plot (Figure~\ref{fig:cactus_plot}). The Ship model maintains a linear, flat trajectory, successfully solving the entire dataset with negligible time variation. Conversely, both the Standard and Improved Cell models exhibit the classic exponential "hockey stick" curve, wherein the final 20\% of computationally hard puzzles cause the branch-and-bound search trees to expand massively, driving up total execution time.

\subsection{Scaling Behaviour and Mathematical Complexity}\label{scaling-behaviour}
To stress-test the models beyond standard parameters, the procedural generator evaluated grid sizes scaling from 36 to 225 total cells. Figure~\ref{fig:line_scaling} maps the average solve times on a logarithmic scale. The Ship model remains an order of magnitude faster across all dimensions. However, it is vital to note that while the Ship model solves a 15x15 grid rapidly, its candidate generation phase becomes heavily memory-bound at this scale, establishing a physical hardware limitation despite its mathematical superiority.

The root cause of the Cell model's exponential scaling is exposed by tracking the Gurobi Branch-and-Bound node count (Figure~\ref{fig:node_complexity}).  The Ship model consistently explores zero nodes—meaning Gurobi's internal presolve and root-node relaxations are powerful enough to identify the optimal Set Packing combination without ever initiating a search tree. The Standard Cell model, however, is forced to explore hundreds of nodes as the grid expands, systematically guessing and backtracking through the dense Big-M matrix.

Crucially, the Improved Cell model tightens the feasible polytope, meaning the solver is forced to explore significantly fewer dead-end branches. This lead to a successful suppression of the node spike in the Standard Cell model.

\subsection{Impact of Valid Inequalities (Constraint Ablation)}\label{subsec:valid-inequalities-impact}
While Figure~\ref{fig:node_complexity} proves that the valid inequalities successfully prune the search tree, this mathematical efficiency does not perfectly translate to a raw reduction in solve times. 

As shown in the Cuts Comparison chart (Figure~\ref{fig:cuts_comparison}), the Improved Cell model only offers marginal speed upgrades, and occasionally performs slightly worse than the Standard model. This exposes the "Overhead vs. Pruning" paradox inherent to Integer Linear Programming.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.7\textwidth]{images/generated_dataset/solver_cuts_comparison_2026-04-27_14-20-56.png}
    \caption{Direct solve time comparison illustrating the matrix overhead introduced by valid inequalities.}
    \label{fig:cuts_comparison}
\end{figure}

The constraint impact analysis (Table~\ref{tab:full_constraint_analysis}) quantifies this speed trade-off. For Medium puzzles, the baseline model (0.1106s) is slower than the full improved model (0.1061s). The valid inequalities successfully intercept bad guesses early, saving time. 

However, on Hard puzzles (0.1432s baseline vs. 0.1472s all constraints), the heuristics are already starved of hints. Forcing Gurobi to process four additional dense equations for every cell introduces matrix bloat (computational overhead) that slightly outweighs the time saved by pruning nodes.

\subsection{The Algorithmic Paradigm Shift: Dense vs. Sparse Matrices}\label{algorithmic-paradigm-shift}
The extreme difference in performance between the two models stems from how spatial constraints are formulated. The Cell-based model relies on dense Big-M transition logic, meaning that Gurobi has to continually cross-reference adjacent decision variables to enforce valid geometry. When the grid dimensions scale, the density of this constraint matrix explodes combinatorially.

The Ship-based model bypasses those issues by abstracting the spatial reasoning out of the ILP matrix entirely. It leverages Python to pre-calculate the valid fleet geometry (Candidate Generation), the problem is reduced to an Edge-Based Set Packing formulation.

The dominance of the Ship model (Figure~\ref{fig:baseline_performance} and Figure~\ref{fig:line_scaling}) conclusively demonstrates that reframing the mathematical paradigm from a localised, dense-matrix state problem, to a global spare-matrix edge-packing problem, yields considerably higher computational returns than micro-optimising a flawed initial model.

\subsection{The Power of Presolve and Root-Node Relaxation}\label{power-of-presolve}
The node complexity analysis (Figure~\ref{fig:node_complexity}) showed that for many instances, the Ship model solved complex instances while only exploring zero or one Branch-and-Bound nodes. This indicates that the Gurobi solver found the mathematical proof of the solution during its pre-processing phase (which consists of matrix reduction and root-node linear relaxation) before initiating a search tree.

This highlights a weakness in the Cell model in that the Big-M formulations yield weak, highly fractional LP bounds.

In contrast, the Set Packing matrices provide incredibly tight LP bounds. The constraints map to the problem well so Gurobi's internal heuristics can instantly deduce the optimal combination, while avoiding the branching that hindered the Cell model.

\subsection{The "Overhead vs. Pruning" Paradox}\label{overhead-vs-pruning}
The ablation study quantifying the impact of the valid inequalities (Figure~\ref{fig:cuts_comparison} and Table~\ref{tab:full_constraint_analysis}) revealed the "Overhead vs. Pruning" paradox inherent to discrete optimisation.

For the Medium heuristic difficulty instances, the valid inequalities proved to be excellent additions as they successfully tightened the feasible polytope. They helped prune the dead-end branches from the search tree which in turn decreased the solve time.

When looking at the Hard difficulty instances, the result is the opposite. The addition of the redundant constraints for every row/column lead to severe matrix bloat. The computational cost of calculating those extra equations during branching, outweighed the time saved by pruning nodes.

This paradox highlights a fundamental trade-off in Integer Linear Programming. Polyhedral tightening is only beneficial to the solver if the time saved via the search-tree pruning exceeds the computational overhead of the expanded constraint matrix.

\subsection{Trade-offs and Comparison with Prior Work}\label{tradeoffs-comparison-prior-work}
While the underlying constraint equations were heavily adapted from the framework proposed by Meuffels \citep{meuffels-battleship}, this project advances that prior work by quantifying the absolute scaling limits of both models under combinatorial explosion.

The evaluation unveiled a strict limitation of the Ship model. Its speed is achieved at the direct expense of spatial complexity. Due to the fact it pre-generates every possible candidate before Gurobi is invoked, memory consumption grows exponentially.  As grids scale towards larger boards, the Candidate Generation phase will eventually lead to a memory overflow. 

Compare this to the Cell model, despite it being a lot slower, it has a much smaller static memory footprint. This establishes a definitive time-space trade-off: the Ship model dominates in processing speed, but the Cell model remains fundamentally more memory-efficient.
